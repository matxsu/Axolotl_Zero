## Projet

**Axolotl Zero** est un dispositif embarqué de cybersécurité offensive développé
from scratch (hardware + firmware) dans le cadre d'un projet annuel ESGI
(Cybersécurité, 2025–2026). Équipe : Ilyes MAJERI & Mathis NGO.

Inspiré du Flipper Zero, il cible **4 vecteurs d'attaque** :

1. **NFC/RFID** (13.56 MHz, PN532) — lecture, dump, clonage de tags
2. **Wi-Fi** (2.4 GHz, ESP32-S3 natif) — scan, deauth, evil twin
3. **BadUSB** (USB HID natif ESP32-S3) — payloads DuckyScript
4. **Sub-GHz** (CC1101, 315/433/868 MHz) — capture/replay de télécommandes

**Cadre strictement pédagogique et légal.** Toutes les démos se font en
laboratoire ESGI sur du matériel appartenant à l'équipe.

## Stack technique

- **MCU** : ESP32-S3 (Xtensa LX7 dual-core, 240 MHz, Wi-Fi 2.4 GHz, USB OTG natif)
- **Langage firmware** : **Rust** (et non C/ESP-IDF — décision actée en v1.1 du CDC)
- **Framework** : `esp-idf-svc` / `esp-idf-hal` (Rust bindings sur ESP-IDF v5.5.3)
- **Build** : `cargo` + `espflash` pour le flash
- **Target** : `xtensa-esp32s3-espidf`
- **Toolchain** : channel `esp` (installé via `espup`)

## Pinout ESP32-S3 (officiel)

| Pin | Rôle | Bus | Notes |
|---|---|---|---|
| GPIO 3 | I²C SDA | I²C0 | PN532 |
| GPIO 4 | I²C SCL | I²C0 | PN532 |
| GPIO 6 | SD CS | SPI | Chip select carte SD |
| GPIO 8 | Display CS | SPI | Chip select écran |
| GPIO 9 | Display DC | GPIO | Data/Command ST7789 |
| GPIO 10 | Display RST | GPIO | Reset écran |
| GPIO 11 | SPI MOSI | SPI | **Partagé display + SD + CC1101** |
| GPIO 12 | SPI SCK | SPI | **Partagé display + SD + CC1101** |
| GPIO 13 | SPI MISO | SPI | **Partagé SD + CC1101** (display write-only) |
| GPIO 14 | CC1101 CS | SPI | Chip select Sub-GHz |
| GPIO 15 | Joystick UP | GPIO (pull-up) | |
| GPIO 16 | Joystick DOWN | GPIO (pull-up) | |
| GPIO 17 | Joystick LEFT | GPIO (pull-up) | |
| GPIO 18 | Joystick RIGHT | GPIO (pull-up) | |
| GPIO 21 | Joystick CENTER | GPIO (pull-up) | |
| GPIO 38 | CC1101 GDO0 | GPIO IRQ | Sub-GHz interrupt |
| GPIO 39 | CC1101 GDO2 | GPIO IRQ | Sub-GHz interrupt |
| GPIO 46 | Display BLK | GPIO | Backlight |
| GPIO 19/20 | USB D- / D+ | USB OTG | BadUSB HID |

> ⚠️ **Règle SPI** : un seul `SpiDriver` partagé avec plusieurs `SpiDeviceDriver`
> (CS différents). NE PAS créer deux `SpiDriver` distincts sur les mêmes pins —
> conflit matériel garanti.

### Chaîne d'alimentation

```
USB-C 5V ──▶ TP4056 (charge LiPo CC/CV 4.2V, 1A max)
                │
                ▼
         LiPo 1S 3.7V 1000 mAh
                │
                ▼
         MT3608 (boost 3.7V → 5V)
                │
                ▼
         LDO 3.3V (AMS1117) ──▶ ESP32-S3 + tous les périphériques
```

- **TP4056** : chargeur linéaire, LEDs status (rouge=charge, bleu=terminé)
- **MT3608** : boost converter step-up, permet de fonctionner sur batterie
- **Autonomie estimée** : ~3-5h en usage mixte (pas de deep sleep implémenté)
- **Budget courant worst case** : ~514 mA (tous modules actifs simultanément)

### État physique

- **Actuellement** : breadboard 830 pts + jumper wires
- **Cible** : boîtier imprimé 3D (PLA), dimensions ~120×70×35 mm
- Les modules breakout (PN532, CC1101, TP4056, MT3608, SD) restent des
  modules du commerce, pas de PCB custom

## Conventions de code Rust

### Gestion d'erreur
- **Bibliothèque** : `anyhow` pour les erreurs remontées jusqu'à `main`
- **Pas d'`unwrap()`** en production (sauf `const`/infaillible documenté)
- **Pas de `panic!`** — préférer `anyhow::bail!` ou `return Err(...)`
- Les drivers retournent `anyhow::Result<T>`, pas des erreurs custom (simplicité)

### Logging
- `log::error!` : erreurs bloquantes
- `log::warn!` : erreurs non-bloquantes (SD absente, carte NFC perdue)
- `log::info!` : événements utilisateur significatifs (boot, carte lue, dump OK)
- `log::debug!` : détails protocole (tentatives de clés, trames I²C)
- **Pas de `println!`** — toujours `log::*`

### Allocations
- Préférer `heapless::Vec`, `heapless::String` aux types standard
- `format!` autorisé pour les chaînes d'UI (usage unique), interdit en hot loop
- Buffers fixes sur la stack > Vec dynamiques

### Naming
- Modules : `snake_case` (`mifare`, `bad_usb`)
- Types publics : `CamelCase` (`Pn532`, `NfcUid`, `MifareDump`)
- Constantes : `SCREAMING_SNAKE_CASE`
- Pins : `GPIO_*` préfixées si constantes

### Commentaires
- Docs publiques : `///` rustdoc sur tout ce qui est `pub`
- Commentaires de module : `//!` en tête de fichier
- Expliquer **pourquoi**, pas **quoi** (le code dit le quoi)

### Règles
- Jamais de commit direct sur `main` ou `dev`
- PR obligatoire `feature/*` → `dev`, puis `dev` → `main` pour les releases
- Messages de commit : format conventionnel `type(scope): message`
  - `feat(nfc): add MIFARE auth with default key dictionary`
  - `fix(spi): resolve SPI2/SPI3 pin conflict`
  - `docs: update pinout in architecture.md`
  - `refactor(ui): extract screens to submodule`

## Pièges connus (à ne pas refaire)

1. **Conflit SPI2/SPI3** — `main.rs` avait deux `SpiDriver` sur les mêmes pins.
   Toujours un seul SPI partagé avec plusieurs `SpiDeviceDriver`.

2. **Re-sélection après échec auth MIFARE** — après `MFAuthent` raté, la carte
   part en état `HALT`. Il FAUT refaire `InListPassiveTarget` avant de retenter
   une autre clé, sinon toutes les suivantes échouent.

3. **`unwrap()` dans les drivers** — interdit, même sur du code "impossible à
   rater". En embarqué, un `panic!` = reboot du device.

4. **`format!` dans une boucle** — alloue sur le heap à chaque tour. Utiliser
   `heapless::String` pré-alloué ou écriture streaming.

5. **Déclaration de pins partagés** — ESP-IDF ne détecte pas toujours les
   conflits de pins au runtime. Vérifier ce fichier avant toute nouvelle
   allocation GPIO.

6. **Patchs `git` dans Cargo.toml** — actuellement mélange `crates.io` et
   `patch.crates-io` sur `esp-idf-*`. Homogénéiser avant la release.

## Conseils pour interagir avec Claude Code sur ce projet

- **Toujours vérifier la branche active** avant de proposer des modifications
  (4 features = 4 branches en parallèle)
- **Respecter le scope d'un module** — pas de code NFC dans `wifi/`, etc.
- **Compiler après chaque ajout** — le cross-compilation Xtensa est lente,
  autant attraper les erreurs tôt
- **Proposer des commits atomiques** — une fonction = un commit, pas des
  méga-commits multi-modules
- **Pour les drivers bas niveau** (PN532, CC1101) : toujours expliquer le
  protocole dans les commentaires de fonction, car c'est de la pédagogie et
  un jury posera des questions
